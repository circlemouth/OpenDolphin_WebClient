# verification-plan.md プレースホルダ埋め方ガイド

> 目的: 残プレースホルダを埋めるために必要な情報と決定事項を整理する。

## 1. 基本情報
- 対象名称: プロジェクト名/システム名/対象バージョン
- 対象範囲: 画面範囲・機能範囲・環境（dev/stage/prod）
- 目的 / 背景: 実施理由（リリース前検証、品質保証、変更影響確認）
- 期間 / マイルストーン: 開始/終了日、レビュー日、承認日
- 参照資料: 正本ドキュメント、設計書、API仕様、運用手順
- 作成日/最終更新日: ドキュメント更新日
- 作成者/承認者: 役割名または担当者名

## 2. 役割と担当切り分け（RACI）
- Responsible: 実作業担当（画面確認/証跡取得）
- Accountable: 最終責任者（承認/判断）
- Consulted: 仕様/運用の相談先
- Informed: 共有先（関係者/管理者）

## 2.1 担当者一覧
- 主担当/副担当/レビュー担当/承認担当/連絡先を具体名で記入

## 3. 確認範囲と除外範囲
- 確認対象: 画面/機能/環境/ユーザー権限
- 除外事項: 今回対象外の範囲（例: Legacy、入院系、外部連携実測）
- 前提条件 / 制約: 利用可能な環境/データ/時間制約

## 4. 画面一覧
- 画面ID/画面名/URL/概要/オーナー/備考
- 画面ID採番ルール（SCR-001 など）
- 画面一覧の正本（AppRouter / 設計書 / 画面遷移図）

## 4.2 画面ごとの確認観点フォーマット
- 各画面の目的/前提/導線/入力/例外/パフォーマンス/権限
- 関連サーバー機能/テストデータ/エビデンスの場所

## 7. 確認手順
- 準備: 環境構築、権限、テストデータ、必要資料
- 実施: 画面単位の確認順序、証跡採取手順
- 結果整理: 指摘分類、重要度判定、対応要否

## 8. チェックリスト/エビデンス
- チェック項目のYes/No/要確認は実施時に記入
- エビデンス保存先/命名規則/保持期間を定義
- 完了条件: 必須項目・証跡・承認の条件

## 9. 運用ルール
- 更新頻度: 定期/トリガー条件
- 版管理: 版番号、変更履歴の場所、旧版保管先

## 12. スケジュール
- フェーズごとの開始/終了/担当

## 13. 変更履歴
- 変更者/承認者を記入

---

## 決定が必要な事項（要確認事項まとめ）
- 対象範囲（画面/機能/環境）と除外範囲の確定
- 画面一覧の正本と採番ルール
- RACI/担当者/承認者
- 確認実施の期間・マイルストーン
- 証跡の保存先/命名規則/保持期間
- 版管理ルール（版番号・履歴・保管先）

## 最終計画書への注記案
- 追加先: `10. 要確認事項` の末尾に「決定が必要な事項」を箇条書きで追記する。
- あるいは `1. 基本情報` 直下に「未確定項目は 10章に集約」と注記を追加する。
